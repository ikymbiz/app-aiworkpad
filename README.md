## 1. システム概要

AIWorkPadは、AI（Google Gemini / Anthropic Claude）を統合したマークダウンエディタである。PWA（Progressive Web App）として構築され、ブラウザ上で動作しつつ、Android端末ではホーム画面に追加してネイティブアプリに近い体験を提供する。

全データはブラウザのIndexedDBに永続保存され、サーバーへのデータ送信はAI API呼び出し時のみ。ユーザーが自分のデータを完全に所有するローカルファースト設計を採用している。

### 1.1 プロダクトビジョン

- Obsidianに匹敵するナレッジマネジメント機能をWebベースで実現する
- AI（Gemini / Claude）によるインライン編集・チャット・ドラフト生成を統合する
- オフラインファースト設計によりデータ主権を確保する
- Android PWAとしてネイティブアプリ同等のUXを実現する

### 1.2 対象ユーザー

- マークダウンで文書作成を行うライター・研究者・エンジニア
- AIを活用して文章のブラッシュアップや調査を行いたいユーザー
- ローカルファーストでデータを自分の端末に保持したいユーザー

### 1.3 バージョン計画

| Ver | ステータス | 概要 |
|-----|----------|------|
| v2.0 | リリース済 | エディタ・ツリー・チャット・AI統合・エクスポート・バックリンク・クイックスイッチャー |
| v3.0 | 現行リリース | Service Workerオフライン・APIキー暗号化(AES-GCM)・SWソース埋め込み |
| v4.0 | 将来構想 | WYSIWYG・グラフビュー・画像添付・マルチターンチャット・クラウド同期・TWA |

## 2. ファイル構成とセットアップ

### 2.1 ファイル一覧

```
AIWorkPad/
├── index.html    … アプリ本体 (HTML/CSS/JS を単一ファイルに統合)
└── sw.js         … Service Worker (オフライン対応)
```

### 2.2 セットアップ手順

1. `index.html` と `sw.js` を同じフォルダに配置する
2. Webサーバー経由で `index.html` を開く（`file://` プロトコルではService Workerが動作しない）
3. 初回起動時に設定モーダルからAPIキー（Gemini / Claude）を入力する
4. APIキーはAES-GCM 256bitで暗号化されてIndexedDBに保存される

### 2.3 簡易起動方法

```bash
# Python3 の場合
cd AIWorkPad/
python3 -m http.server 8080

# Node.js の場合
npx serve .
```

ブラウザで `http://localhost:8080` を開く。

### 2.4 sw.js 未配置時の動作

`sw.js` が同一ディレクトリに存在しない場合、`index.html` 内に埋め込まれたSWソースコードから `sw.js` を自動生成し、画面下部にダウンロードバナーを表示する。ユーザーがダウンロードボタンをクリックし、`index.html` と同じフォルダに配置した後、ページを再読み込みすればService Workerが有効化される。

## 4. データアーキテクチャ

### 4.1 IndexedDB ストア構成

全データはブラウザのIndexedDB上に永続保存される。localforageライブラリを使用し、非同期キー値ストアとして8つの独立したストアを管理する。DB名はすべて `AIWorkPad`。

| ストア名 | キー | 値の型 | 用途 |
|---------|-----|-------|------|
| `docList` | `'list'` | Array | ドキュメントメタデータ一覧。フォルダツリー構造を定義する |
| `document` | ドキュメントID | Object | ドキュメント本文。`{ id, content, lastModified }` |
| `history` | バージョンID | Object | バージョン履歴。`{ id, docId, timestamp, content, reason, description }` — 各doc最大5件 |
| `chat` | メッセージID | Object | AIチャット履歴。`{ id, docId, timestamp, role, content }` — ドキュメント別に管理 |
| `settings` | `'appSettings'` | Object | アプリ設定。APIキー(暗号化済)、デフォルトモデル、通知設定、ペルソナ一覧 |
| `settings` | `'pendingAI'` | Object | 中断されたAI処理タスク。レジューム用 |
| `settings` | `'crypto_salt'` | Array | 暗号化用ソルト (16バイト乱数)。初回起動時に生成 |
| `bookmarks` | `'list'` | Array | ブックマーク済みドキュメントIDの配列 |
| `templates` | `'list'` | Array | ユーザー定義テンプレート。`{ id, name, content }` |

### 4.2 フォルダ構造データモデル

ドキュメントは親子関係によりツリー構造を形成する。`docList` ストアに格納されるメタデータに `parentId` フィールドを持ち、`null` の場合ルート階層となる。最大階層深度は10に制限される。

#### ドキュメントメタデータ構造

| `id` | string | 一意識別子。`"doc_" + Date.now()` 形式 |

| `isExpanded` | boolean | ツリー表示時のフォルダ開閉状態 |
| `updatedAt` | number | 最終更新タイムスタンプ (`Date.now()`) |

### 4.3 ドキュメント本文構造

| フィールド | 型 | 説明 |

## 5. セキュリティ要件 — APIキー暗号化

### 5.1 暗号化方式

Web Crypto APIの `SubtleCrypto` インターフェースを使用し、AES-GCM (256bit) でAPIキーを暗号化する。

| 項目 | 仕様 |
|------|------|
| 暗号アルゴリズム | AES-GCM (256bit) |
| 鍵導出 | PBKDF2 (SHA-256, 100,000回イテレーション) |
| ソルト | 初回起動時に `crypto.getRandomValues()` で16バイト生成、IndexedDB (`settings` ストア、キー `crypto_salt`) に永続保存 |
| IV (初期化ベクトル) | 暗号化ごとに12バイトのランダムIVを生成 |
| 保存形式 | Base64エンコード: `[12byte IV] + [暗号文]` を連結したArrayBufferをBase64化 |
| 対象データ | Gemini APIキー (`geminiKey`)、Claude APIキー (`claudeKey`) |
| 鍵の派生元 | デバイス固有フィンガープリント: `navigator.userAgent + screen.width + 'x' + screen.height + screen.colorDepth + timezoneOffset + navigator.language + navigator.hardwareConcurrency` をPBKDF2で鍵導出。ユーザーがパスワードを入力する必要はない |

### 5.2 暗号化フロー

#### APIキー保存時

1. ユーザーが設定モーダルで平文APIキーを入力
2. デバイス固有フィンガープリントを生成
3. IndexedDBからソルトを取得（初回は生成）
4. PBKDF2でCryptoKeyを導出
5. `crypto.getRandomValues()` で12バイトIVを生成
6. AES-GCMで平文キーを暗号化
7. `IV + 暗号文` をBase64化してIndexedDBに保存

#### APIキー使用時 (AI API呼び出し)

1. IndexedDBからBase64暗号文を読み出し
2. Base64デコード → IV (先頭12バイト) と暗号文を分離
3. PBKDF2で同じCryptoKeyを導出
4. AES-GCMで復号 → 平文APIキーを取得
5. 平文キーをAPI呼び出しに使用（メモリ上のみ、永続化しない）

#### 設定モーダル表示時

1. 暗号化されたキーをIndexedDBから読み出し
2. AES-GCMで復号
3. 平文キーを `<input type="password">` フィールドに表示

#### JSONバックアップ時

- APIキーは暗号化状態のままエクスポートされる
- 別デバイスへのインポート時は復号不可（デバイス固有フィンガープリントが異なるため）
- セキュリティ上の意図的な設計。インポート先で再入力が必要

### 5.3 既存データの自動マイグレーション

`loadSettings()` 実行時に、保存済みのAPIキーが平文かどうかを `isEncrypted()` で判定する。判定ロジックは以下の通り:

- `AIzaSy` で始まる → Gemini平文キー → 暗号化して上書き保存
- `sk-` で始まる → Claude平文キー → 暗号化して上書き保存
- Base64デコードが成功し、長さが30文字を超える → 暗号化済みとみなす

v2.0以前から移行するユーザーは初回起動時に自動で暗号化が適用される。

### 5.4 その他のセキュリティ要件

| 項目 | 仕様 |
|------|------|
| 通信プロトコル | AI API呼び出しはHTTPS必須 |
| CORS | Claude API: `anthropic-dangerously-allow-browser` ヘッダーを使用（ブラウザ直接呼び出し） |
| XSS対策 | marked.jsのデフォルト設定によるサニタイズ |
| データ保管 | 全データはローカルのみ。サーバーへのデータ送信はAI API呼び出し時のみ |

## 6. Service Worker — オフライン対応

### 6.1 アーキテクチャ

Service Worker (`sw.js`) は、ブラウザの仕様上 `index.html` とは別ファイルとして配置する必要がある。ただし、`index.html` 内の `<script id="sw-source" type="text/sw-template">` タグにSWのソースコードが完全に埋め込まれており、`sw.js` が未配置の場合はユーザーにダウンロードを案内する仕組みを備える。

### 6.2 キャッシュ戦略

| キャッシュ名 | 戦略 | 対象 |
|-------------|------|------|
| `aiworkpad-v3.0-shell` | Network First + Cache Fallback | `index.html` およびナビゲーションリクエスト。オンライン時は最新版を取得してキャッシュを更新、オフライン時はキャッシュから配信 |
| `aiworkpad-v3.0-cdn` | Cache First + Network Fallback | CDNリソース (Tailwind CSS, FontAwesome, localforage, marked.js, JSZip)。初回アクセス時にキャッシュし、以降はキャッシュから配信 |
| (キャッシュなし) | Network Only | AI API呼び出し (Gemini API, Claude API)。キャッシュしない |

### 6.3 キャッシュ対象リソース

#### App Shell (必須キャッシュ)

- `./` (ルート)
- `./index.html`

#### CDN リソース (Best-effort キャッシュ)

- `https://cdn.tailwindcss.com?plugins=typography`
- `https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.4.0/css/all.min.css`
- `https://cdnjs.cloudflare.com/ajax/libs/localforage/1.10.0/localforage.min.js`
- `https://cdn.jsdelivr.net/npm/marked/marked.min.js`
- `https://cdnjs.cloudflare.com/ajax/libs/jszip/3.10.1/jszip.min.js`

#### FontAwesome Webfont (動的キャッシュ)

- `cdnjs.cloudflare.com/ajax/libs/font-awesome` にマッチするリクエストを動的にキャッシュ

### 6.4 ライフサイクルイベント

#### Install

- App Shellリソースをプリキャッシュ
- CDNリソースをbest-effortでプリキャッシュ（失敗してもインストールは成功する）
- `self.skipWaiting()` で即座にアクティベーション

#### Activate

- `CACHE_VERSION` (`aiworkpad-v3.0`) プレフィックスを持たない古いキャッシュをすべて削除
- `self.clients.claim()` で即座にページを制御

#### Fetch

1. AI API (`generativelanguage.googleapis.com`, `api.anthropic.com`) → Network Only
2. 非HTTPプロトコル (`chrome-extension://` 等) → スキップ
3. CDN / FontAwesome → Cache First (キャッシュヒットならキャッシュから返却、なければネットワーク取得してキャッシュ)
4. HTMLナビゲーション → Network First (オンライン時に最新を取得してキャッシュ更新、オフラインならキャッシュフォールバック)
5. その他の静的リソース → Cache First

### 6.5 バージョン更新フロー

1. 新しい `sw.js` がサーバーにデプロイされる
2. ブラウザが既存SWとの差分を検出し、新SWを `installing` 状態に遷移
3. 新SWが `installed` 状態になると `updatefound` イベントが発火
4. `index.html` 側のリスナーが `confirm()` ダイアログでユーザーに更新を確認
5. ユーザーが承認すると `skipWaiting` メッセージを送信し、ページを再読み込み

### 6.6 sw.js 未配置時の自動生成フロー

1. `index.html` 起動時に `fetch('./sw.js', { method: 'HEAD' })` でファイル存在を確認
2. 404の場合、`<script id="sw-source">` タグ内のSWソースコードを取得
3. `Blob` + `URL.createObjectURL()` でダウンロード可能なURLを生成
4. 画面下部に青いバナーを表示: 「オフライン対応には sw.js が必要です」+ ダウンロードボタン
5. ダウンロードクリック後、「index.htmlと同じフォルダに配置後、再読み込みしてください」の案内に切り替わる
6. ✕ボタンでバナーを閉じることも可能（SWなしでも動作する）

- **AIアクションシート**: テキスト選択時にキャレット座標付近にフローティング表示。6アクションの横スクロールカルーセル
- **ステータスバー**: 文字数 / 単語数 / 行数 / タグ数 / ブックマーク★

| 定期保存間隔 | 5分ごとにバージョン自動保存 (`SAVE_INTERVAL = 300000ms`) |
| AI処理 | Web Worker上でAPI呼び出しを実行し、メインスレッドのブロッキングを回避 |
| DOM操作 | ドキュメントツリーは差分ではなく全再描画。数百件規模では問題ない設計 |
| 検索デバウンス | 全文検索は入力300ms後にデバウンス実行 |

### 14.2 データ永続性

| 項目 | 要件 |
|------|------|
| ストレージ | IndexedDB (localforage) — ブラウザのストレージクォータに依存 (通常 数百MB～数GB) |
| バージョン履歴 | 各ドキュメント最大5件。超過分は古いものから自動削除 |
| データ所有権 | 全データはユーザーの端末ローカルに保存。サーバーへのデータ送信はAI API呼び出し時のみ |
| バックアップ | JSON / ZIP一括エクスポートにより手動バックアップ可能 |
| オフライン | Service Worker によりApp ShellとCDNリソースをキャッシュ。オフラインでもアプリ起動と編集が可能 (AI機能はオンライン必須) |

### 14.3 セキュリティ

| 項目 | 要件 |
|------|------|
| APIキー保管 | AES-GCM 256bit + PBKDF2で暗号化してIndexedDBに保存 |
| 通信 | AI API呼び出しはHTTPS経由 |
| CORS | Claude API: `anthropic-dangerously-allow-browser` ヘッダー使用 (ブラウザ直接呼び出し) |
| 入力サニタイズ | marked.jsのデフォルト設定によるXSS軽減 |
| バックアップ | JSONエクスポート時、APIキーは暗号化状態を維持。別デバイスでは復号不可 |

### 14.4 互換性

| 項目 | 要件 |
|------|------|
| 対応ブラウザ | Chrome 80+, Firefox 78+, Safari 14+, Edge 80+ (IndexedDB + Web Worker + Web Crypto API対応が必須) |
| モバイル | Android Chrome (PWAホーム画面追加対応)、iOS Safari (PWA制限あり) |
| 画面サイズ | 320px以上の画面幅に対応。100dvh使用でモバイルキーボード表示時のレイアウト崩れを防止 |

### 14.5 PWA要件

| mobile-web-app-capable | Android Chrome / iOS Safari でのホーム画面追加に対応 |

## 17. API仕様

### 17.1 Gemini API

| 項目 | 値 |
|------|-----|
| エンドポイント | `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent` |
| 認証 | URLパラメータ `key=` でAPIキーを指定 (復号後の平文キーを使用) |
| HTTPメソッド | POST |
| Content-Type | `application/json` |
| リクエストボディ | `{ contents: [{ role: 'user', parts: [{ text: prompt }] }], systemInstruction: { parts: [{ text: systemPrompt }] } }` |
| レスポンス | `{ candidates: [{ content: { parts: [{ text: resultText }] } }] }` |

### 17.2 Claude API

| 項目 | 値 |
|------|-----|
| エンドポイント | `https://api.anthropic.com/v1/messages` |
| 認証 | `x-api-key` ヘッダーに復号後の平文キーを指定 |
| 必須ヘッダー | `anthropic-version: 2023-06-01`, `anthropic-dangerously-allow-browser: true`, `content-type: application/json` |
| HTTPメソッド | POST |
| リクエストボディ | `{ model: 'claude-3-haiku-20240307', max_tokens: 4000, system: systemPrompt, messages: [{ role: 'user', content: prompt }] }` |
| レスポンス | `{ content: [{ text: resultText }] }` |

### 17.3 Web Crypto API (暗号化)

| 項目 | 値 |
|------|-----|
| 鍵導出 | `crypto.subtle.importKey('raw', fingerprint, 'PBKDF2', false, ['deriveKey'])` → `crypto.subtle.deriveKey({ name: 'PBKDF2', salt, iterations: 100000, hash: 'SHA-256' }, keyMaterial, { name: 'AES-GCM', length: 256 }, false, ['encrypt', 'decrypt'])` |
| 暗号化 | `crypto.subtle.encrypt({ name: 'AES-GCM', iv: 12bytes }, key, plaintext)` |
| 復号 | `crypto.subtle.decrypt({ name: 'AES-GCM', iv: 12bytes }, key, ciphertext)` |
| IV生成 | `crypto.getRandomValues(new Uint8Array(12))` |
| ソルト生成 | `crypto.getRandomValues(new Uint8Array(16))` |
